Le Porte
Il meccanismo che permette a più applicazioni di usare la rete contemporaneamente sullo stesso indirizzo IP.
Il problema: un indirizzo IP non basta

Ogni host ha un unico indirizzo IP che lo identifica nella rete. Ma un computer esegue decine di processi contemporaneamente: il browser su più schede, un client email, un gioco online, un client SSH. Tutti questi processi ricevono e inviano dati attraverso lo stesso indirizzo IP. Come fa il sistema operativo a sapere a quale processo consegnare un determinato pacchetto ricevuto?

La risposta sono le porte: un numero a 16 bit (0–65535) che identifica il processo destinatario (o mittente) all'interno di un host. Indirizzo IP + porta formano un socket, l'endpoint univoco di una comunicazione di rete.

Socket: la coppia (IP, porta) che identifica univocamente un endpoint

Un socket è la combinazione di un indirizzo IP e un numero di porta: 192.168.1.50:52341. Una connessione TCP è identificata dalla quadrupla:

(IP sorgente, porta sorgente, IP destinazione, porta destinazione)

Questa quadrupla deve essere univoca per ogni connessione attiva sul sistema. Grazie a questo, lo stesso server web su porta 80 può gestire migliaia di connessioni contemporanee: ogni connessione differisce almeno per la porta sorgente del client.

Esempio reale:

Browser → server web: (192.168.1.50:52341, 93.184.216.34:443) Browser → altro sito: (192.168.1.50:52342, 151.101.0.81:443) SSH client → server: (192.168.1.50:55100, 10.0.0.1:22) Mail client → IMAP: (192.168.1.50:49201, 93.184.10.5:993)

Porte stateful: TIME_WAIT e la questione del riuso

Quando una connessione TCP si chiude, la porta client non viene rilasciata immediatamente. Il sistema operativo mantiene la connessione nello stato TIME_WAIT per un periodo di 2×MSL (Maximum Segment Lifetime), solitamente 60–120 secondi. Il motivo: se l'ACK finale venisse perso, il server reinvierebbe il FIN e il client deve poter rispondere correttamente. Durante TIME_WAIT quella quadrupla (IP, porta, IP, porta) non può essere riusata.

In server ad alto traffico (es. load balancer) questo può esaurire le porte efimere disponibili. Soluzioni: SO_REUSEADDR / SO_REUSEPORT, aumento del range efimero, tuning del tcp_tw_reuse del kernel Linux.

NAT e le porte: come il router gestisce migliaia di connessioni

Il router NAT mantiene una tabella di traduzione che associa ogni connessione interna (IP privato : porta sorgente) a una coppia (IP pubblico : porta NAT). Quando due host interni usano la stessa porta efimera, il router NAT assegna porte pubbliche diverse per distinguerle. Questo è chiamato PAT (Port Address Translation) o NAPT.

Controllo degli Errori in TCP
Checksum, numeri di sequenza, ACK, timeout, ritrasmissione, SACK: come TCP garantisce che ogni byte arrivi corretto e nell'ordine giusto.
Perché serve il controllo degli errori?

IP è un protocollo best-effort: consegna i pacchetti senza garanzie. Un pacchetto può andare perso (router congestionato che scarta), arrivare corrotto (errore sul link fisico), arrivare duplicato, o arrivare fuori ordine. TCP deve rilevare e correggere tutti questi problemi prima di consegnare i dati all'applicazione. I meccanismi sono quattro, che lavorano insieme:

  • Checksum: rileva corruzione dei dati in transito
  • Numeri di sequenza: rileva perdite, duplicati, riordina fuori ordine
  • Acknowledgment (ACK): conferma la ricezione, segnala cosa manca
  • Timeout e ritrasmissione: recupera i segmenti persi
1 — Il Checksum TCP: come funziona davvero
Algoritmo del checksum (complemento a 1)

Il checksum TCP copre: lo pseudo-header IP (IP sorgente + IP destinazione + protocollo 6 + lunghezza TCP), l'header TCP, e il payload. L'algoritmo:

  1. Il campo checksum dell'header viene azzerato prima del calcolo
  2. Tutti i dati vengono trattati come word da 16 bit e sommate
  3. I carry overflow vengono riportati (addizionati al bit meno significativo — wrap around)
  4. Il risultato viene invertito bit a bit (complemento a 1): questo è il checksum
  5. In ricezione: si rifà la stessa somma incluso il checksum ricevuto. Se tutti i bit del risultato sono 1 (0xFFFF), il dato è integro. Se no, il segmento viene scartato silenziosamente.
Esempio numerico — calcolo checksum su dati semplici
Somma complemento a 1 — esempio con 3 word da 16 bit
Word 1 (dati):0100 1000 0110 0101= 0x4865 = "He"
Word 2 (dati):0110 1100 0110 1100= 0x6C6C = "ll"
Word 3 (dati):0110 1111 0000 0000= 0x6F00 = "o\0"
Somma:1 0001 0101 1111 1011= 0x115FB (17 bit!)
Wrap carry:0001 0101 1111 11000x15FB + 0x0001 = 0x15FC
Complemento a 1:1110 1010 0000 0011= 0xEA03 — questo è il checksum
Verifica in ricezione: somma delle 3 word + checksum ricevuto = 0xFFFF → dato integro ✓
Il checksum TCP è necessario ma non sufficiente: due errori opposti sullo stesso bit possono annullarsi e passare il controllo. Per questo i layer superiori (TLS, applicazioni) spesso aggiungono CRC o hash crittografici più robusti. Il checksum TCP rileva la grande maggioranza degli errori casuali di trasmissione.
2 — Numeri di sequenza e ACK: la meccanica della conferma
Come funzionano i numeri di sequenza

TCP tratta i dati come un flusso continuo di byte, non come messaggi separati. Il Sequence Number nell'header TCP indica la posizione nel flusso del primo byte di payload contenuto in quel segmento. Se il segmento trasporta 500 byte e ha Sequence Number = 1000, il prossimo segmento avrà Sequence Number = 1500.

Il valore iniziale (ISN) è scelto casualmente al momento della connessione (SYN) per motivi di sicurezza: impedisce che un attaccante possa iniettare segmenti indovinando il numero di sequenza corrente.

L'Acknowledgment Number contiene il valore del prossimo byte atteso dal ricevitore. Un ACK = 1501 significa: "ho ricevuto correttamente tutti i byte fino al 1500 incluso, aspetto il byte 1501". È un ACK cumulativo: conferma tutto fino a quel punto, senza indicare cosa è arrivato dopo.

RTO vs Fast Retransmit: quando si usa quale
  • RTO (Retransmission TimeOut): scatta quando non arriva nessun ACK entro il timeout calcolato. Il mittente deduce che il segmento si è perso o il destinatario è irraggiungibile. L'RTO è adattivo: calcolato con la formula di Jacobson/Karels basata sul RTT misurato (RTO = SRTT + 4×RTTVAR). Quando scatta, TCP dimezza la congestion window e ritrasmette il segmento più vecchio non confermato. Reazione drastica.
  • Fast Retransmit (3 dup-ACK): tre ACK duplicati per lo stesso sequence number sono un forte segnale che quel segmento è perduto ma la rete funziona (i segmenti successivi arrivano). TCP ritrasmette subito senza aspettare l'RTO. Reazione meno drastica: cwnd dimezzata ma non azzerata (Fast Recovery).

Recupero perdita grave

Fast Retransmit
Recupero perdita rapido

Segnalazione duplicati

Estensione di SACK che segnala anche i segmenti ricevuti in duplicato. Il mittente può così capire se la rete sta replicando pacchetti o se i suoi RTO sono troppo aggressivi.

Il Three-Way Handshake TCP
Non solo SYN/SYN-ACK/ACK. Cosa negozia davvero, come funzionano gli ISN, perché è a 3 vie e non 2, gli stati della macchina a stati, e tutto quello che succede sotto.
Perché serve un handshake prima di trasmettere?

TCP garantisce affidabilità attraverso numeri di sequenza. Prima di trasmettere dati, i due host devono sincronizzare i propri numeri di sequenza iniziali (ISN — Initial Sequence Number). Senza questa sincronizzazione il ricevitore non saprebbe come ordinare i dati in arrivo né come generare gli ACK corretti.

Oltre alla sincronizzazione degli ISN, il three-way handshake negozia anche: la MSS (Maximum Segment Size), il supporto a SACK, la Window Scale, il supporto a Timestamps, e altri parametri. È la fase di setup completa della connessione.

Perché proprio 3 passi e non 2?

Con 2 passi (SYN + SYN-ACK), solo il server conferma l'ISN del client. Il client non conferma l'ISN del server: il server non sa se il suo SYN-ACK è arrivato e non può essere sicuro che il client sia pronto a ricevere. Il terzo passo (ACK del client) serve a:

  1. Confermare che il client ha ricevuto l'ISN del server
  2. Dare al server la certezza che la connessione è bidirezionale e funzionante
  3. Permettere al server di allocare le risorse solo quando c'è effettiva certezza di connessione (protezione da SYN flooding)

Un 4-way handshake sarebbe ridondante: il terzo passo (ACK del client all'ISN del server) è sufficiente per entrambe le conferme rimanenti.

Gli ISN: perché casuali e come vengono scelti
ISN casuali — sicurezza della connessione TCP

Gli ISN non sono mai 0. Vengono scelti pseudo-casualmente dal sistema operativo, con un valore che cambia nel tempo. Il motivo è sicurezza:

  • Prevenzione TCP Hijacking: se gli ISN fossero prevedibili, un attaccante potrebbe iniettare segmenti fasulli in una connessione altrui fingendo di essere il client legittimo. Con ISN casuali, sarebbe impossibile indovinare il numero di sequenza corrente.
  • Prevenzione di connessioni zombie: segmenti di connessioni precedenti terminati male (con lo stesso (IP, porta)) potrebbero essere accettati da una nuova connessione se avessero sequence number coincidenti. ISN casuali rendono questo quasi impossibile.
  • Protezione da attacchi blind spoofing: un attaccante che non vede il traffico non riesce a costruire pacchetti con sequence number validi.

L'RFC 6528 raccomanda di calcolare l'ISN come ISN = M + F(localip, localport, remoteip, remoteport, secretkey) dove M è un timer che incrementa ogni 4 µs e F è una funzione hash con chiave segreta.

Le opzioni TCP negoziate nell'handshake
MSS — Maximum Segment Size
Dimensione massima del payload

Ogni host dichiara la propria MSS: la dimensione massima del payload TCP che è disposto a ricevere in un singolo segmento. Di solito MTU - 40 byte (20 IP + 20 TCP). Su Ethernet: MTU=1500, MSS=1460. La MSS effettiva della connessione è il minimo tra le due. Previene la frammentazione IP: se i segmenti hanno dimensione ≤ MSS, entrano in un singolo frame Ethernet senza frammentazione.

Window Scale (WS)
Finestra oltre 65535 byte

Il campo Window Size nell'header TCP è a 16 bit: massimo 65535 byte. Su connessioni ad alta latenza e alta banda (intercontinentali, fibra), questa finestra è un collo di bottiglia. L'opzione Window Scale (RFC 7323) estende la finestra moltiplicandola per un fattore 2^n (n ∈ 0–14). Negoziata nell'handshake, valida per tutta la connessione. Entrambi i lati devono supportarla affinché abbia effetto.

Timestamps (TS)
RTT preciso e protezione PAWS

Ogni segmento include un timestamp. Il ricevitore riflette il timestamp del mittente nel campo echo-reply. Il mittente calcola il RTT esatto per ogni segmento. Secondo utilizzo: PAWS (Protection Against Wrapped Sequences) — i timestamp permettono di scartare segmenti con Sequence Number riutilizzato da connessioni precedenti (wrap-around dei 32 bit di SEQ).

SACK-Permitted
Abilita ACK selettivi

Se entrambi i lati includono l'opzione SACK-Permitted nel SYN/SYN-ACK, la connessione supporta Selective Acknowledgment. Senza questo flag durante l'handshake, SACK è disabilitato per tutta la connessione, anche se i sistemi lo supportano in astratto. Oggi praticamente tutti i sistemi moderni lo negoziano sempre.

La macchina a stati TCP — gli stati dell'handshake
TCP State Machine — apertura connessione
Lato client (active open)
  1. CLOSED → chiamata a connect() → invia SYN → SYN_SENT
  2. SYN_SENT → riceve SYN-ACK → invia ACK → ESTABLISHED
Lato server (passive open)
  1. CLOSED → chiamata a bind() + listen() → LISTEN
  2. LISTEN → riceve SYN → invia SYN-ACK → SYN_RECEIVED
  3. SYN_RECEIVED → riceve ACK → ESTABLISHED → chiamata a accept() sblocca l'applicazione
La SYN queue e l'accept queue (e il SYN flooding)

Quando un server riceve un SYN e risponde con SYN-ACK, la connessione viene inserita in una SYN queue (half-open connections queue). Solo dopo aver ricevuto il terzo ACK la connessione viene spostata nell'accept queue (fully established), pronta per essere restituita all'applicazione tramite accept().

Un attacco SYN flood invia migliaia di SYN con IP sorgente falsi: il server risponde con SYN-ACK a indirizzi inesistenti, riempie la SYN queue, e non riesce più ad accettare connessioni legittime. Contromisura: SYN cookies — il server non alloca risorse al SYN ma codifica lo stato nella cookie del SYN-ACK, verificando la legittimità solo quando arriva l'ACK finale.

Simultaneous Open: due host aprono contemporaneamente

TCP gestisce anche il caso raro in cui entrambi i lati inviano un SYN contemporaneamente alla stessa coppia di porte. Entrambi entrano nello stato SYN_SENT, poi ricevono il SYN dell'altro e entrano in SYN_RECEIVED, poi inviano SYN-ACK. Alla fine si stabilisce una singola connessione con 4 messaggi invece di 3. Raramente usato nella pratica ma definito nella RFC 793.

Chiusura della connessione — il four-way teardown
Come si chiude una connessione TCP

TCP è full-duplex: ogni direzione del flusso deve essere chiusa separatamente. La chiusura ordinata richiede 4 messaggi (o 3 se i FIN vengono aggregati).

Four-way teardown — chiusura ordinata
Client → Server: FIN SEQ=X, FLAGS=FIN+ACK (client non ha più dati da inviare) Client stato: FIN_WAIT_1 Server → Client: ACK ACK=X+1 (server ha ricevuto il FIN del client) Client stato: FIN_WAIT_2 (il server può ancora inviare dati in questa fase — half-close) Server → Client: FIN SEQ=Y, FLAGS=FIN+ACK (server ha finito anche lui) Client stato: TIME_WAIT Client → Server: ACK ACK=Y+1 (conferma il FIN del server) Client rimane in TIME_WAIT per 2×MSL (~60-120 secondi) poi → CLOSED
RST — chiusura abortiva

Invece del FIN, un host può inviare un segmento con flag RST: la connessione viene terminata immediatamente e brutalmente, senza il four-way handshake. Il RST non aspetta risposta. Viene usato in caso di errori gravi, connessioni non riconosciute (es. un SYN arriva su una porta chiusa), timeout applicativi (SO_LINGER = 0), o per rifiutare connessioni indesiderate. Non c’è TIME_WAIT dopo un RST.

Punti chiave da ricordare all'orale: (1) i 3 passi sincronizzano ISN in ENTRAMBE le direzioni; (2) le opzioni TCP vengono negoziate nel SYN/SYN-ACK e valgono per tutta la connessione; (3) il SYN conta come 1 byte virtuale nei numeri di sequenza; (4) la randomizzazione degli ISN previene TCP hijacking; (5) il server entra in LISTEN prima dell'handshake, non in CLOSED.
